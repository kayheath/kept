# Addendum — Kept

Depth that supports the brief but doesn't belong inside it. For downstream work (PRD, architecture, build planning). The brief describes the *finished gift*; this captures *how it gets built* and *where the detail lives*.

## Build Order (internal sequencing — NOT a delivery plan)

He receives one complete, finished app — there is no phased rollout *to him*. But Kayla is the sole builder (using BMad), and intends to build it in this sequence:

1. **Core — "calm + you":** the calm home screen (one focus + growing done-pile + gentle today-scoped peek, bucketed by time of day), frictionless capture with auto-bucketing, and the note vault surfacing unconditional messages at the right moments.
2. **Real-life structure:** the two-date model (deadline vs. do-date), the day-coach / night-soft-landing tone shift, and the empty-day invitation.
3. **Moments of delight:** the slow completion ritual and the day journal.

This ordering exists so the emotional core is provably working before the surrounding structure is layered on. It is build sequencing only — the brief's "one finished product" framing is the contract for what he receives.

## Technical Direction

- **Platform:** desktop application.
- **Architecture:** offline-first / local-first. All data lives on his computer.
- **No backend:** no accounts, no cloud, no sync, no live connection. Her presence is pre-written and saved to disk, so it never depends on her being reachable or online.
- Stack to be decided during PRD/architecture; the only hard constraint is fully local, no server dependency.

## The Design Law (load-bearing — repeated here so it's never lost)

> **The love in Kept is never conditional, never withheld, never earned.**

Every feature, copy choice, and interaction must be checked against this. It is the one rule that, if violated, would do real harm rather than merely miss the mark.
